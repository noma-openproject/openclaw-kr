# Noma 중간점검 Addendum (2026-03-27)

## 목적
- 최근 7일 기준 OpenClaw 공식 릴리스, GitHub 이슈, 보안 권고, 한국/해외 커뮤니티 팁을 반영해 Noma의 현재 위치를 다시 점검한다.
- 제안서(v3.2.2)와 프로젝트 지식 문서에 바로 반영할 redline을 정리한다.

## 1. 이번 주 결론
1. Noma는 아이디어 검증 단계가 아니라 **Phase 0 proof-of-life를 이미 통과한 상태**다.
2. 이제 병목은 “설치”보다 **운영 신뢰성**이다.
3. Phase 1 우선순위는 기능 확장이 아니라 아래 4개다.
   - 안정적 relay / named tunnel
   - 실행 영수증 + 비용 가시성 + 관측성
   - Known-good 버전 관리 / upgrade guard
   - Windows smoke-test 및 배포 경로
4. 카카오는 이미 round-trip이 성공했으므로, 다음 단계는 “연동됨”을 더 증명하는 것이 아니라 **안정성과 복구력**을 높이는 것이다.
5. KR skill pack 확장은 계속 필요하지만, 브라우저-heavy 자동화보다 **API-first / local-first 레시피**를 더 우선해야 한다.

## 2. 현재 상태 재판정
### 이미 검증된 것
- OpenClaw baseline + ChatGPT OAuth + Electron launcher는 검증 완료.
- 카카오 채널 + 챗봇 + 스킬 서버 + callback round-trip까지 성공.
- alpha-secure 철학(ask_before_execute, install deny, image/agents.create 차단, Kakao Safe Mode)은 방향이 맞다.

### 아직 비어 있는 것
- 비용 가시성 UI
- 실행 상태/실행 영수증 UI
- named tunnel 또는 클라우드 relay
- Windows smoke-test
- embeddings/memory search의 실제 동작 확인
- 업그레이드 전후 capability audit

## 3. 이번 주 외부 Delta (반영 필요)
### 공식 릴리스 / 공식 표면
- OpenClaw 2026.3.24 릴리스:
  - `/v1/models`, `/v1/embeddings` 추가
  - “Available Right Now” 도구 가시성 강화
  - skill install recipe / status-filter / key setup hint 강화
  - Docker/containers, sequential channel boot 등 운영면 개선
- 시사점:
  - Noma는 launcher에서 “지금 실제로 쓸 수 있는 도구”를 별도 surface로 더 잘 보여줄 수 있다.
  - 스킬 설치 UX와 키 설정 UX는 Noma가 흡수하기 좋은 개선 포인트다.

### 이번 주 보안 권고(3/26)
- gateway/admin scope, reconnect, session kill, history route, plugin fallback deleteSession 등 다수 권고가 같은 날 공개.
- 시사점:
  - Noma의 trust layer는 optional이 아니라 core다.
  - alpha-secure를 더 강하게 유지해야 하며, public-facing remote surface는 최소 권한 원칙을 더 엄격히 적용해야 한다.

### 오늘/어제 GitHub 이슈 패턴
- Browser MCP renderer process leak → 장시간 실행 시 메모리 누수/OOM
- memoryFlush projection bug → compaction/memory flush가 늦게 발동
- UI usage=0 / 실제 토큰 사용 누락
- `/model` override가 sub-agent runtime에 반영되지 않는 이슈
- shared hook fallback이 메인 세션에 소음(system event) 주입
- macOS 3.24 browser tool regression
- 시사점:
  - Noma는 “예쁘게 감싸는 launcher”를 넘어서 **운영 guard**를 제공해야 한다.
  - launcher / relay / receipt / audit 쪽 가치가 더 커졌다.

### 한국 커뮤니티(최근 7일)
- 기억보다 **운영 안정성, 실제 확인, 실패 복구**를 우선한다는 흐름이 강해짐.
- 설치 자체보다 “무겁고 버그가 많아서 직접 인터페이스/셸을 만든다”는 반응이 나옴.
- 멀티 AI 역할 분리, 콘텐츠 자동화, 카카오/텔레그램 결합 실험이 늘고 있음.
- 시사점:
  - Noma의 포지션은 여전히 유효하다.
  - 다만 “쉽게 설치”보다 “계속 믿고 돌릴 수 있음”을 전면 가치로 올려야 한다.

### 해외 커뮤니티(최근 7일)
- observability 대시보드(ClawMetry류), plugin-based model routing/Composio/knowledge graph memory, gateway audit 등이 화두.
- 비용 최적화 팁은 공통적으로 아래로 수렴:
  - 작은 모델을 routine task에 사용
  - bootstrap files/skills를 짧게 유지
  - heartbeat 비용 통제
  - memory/local embeddings 전략
- 시사점:
  - Noma Phase 1에 **Token Saver + Execution Receipts + Observability-lite**를 넣어야 한다.

## 4. 제안서(v3.2.2)에 바로 넣을 redline
### A. 문서 상태 동기화
현재 제안서는 구현 기성이 0% 기준으로 남아 있다.
아래처럼 바꾼다.
- Phase 0 전체 기성: 31% → **~75% 이상 (proof-of-life, Kakao callback, launcher POC, KR skill 1개, alpha-secure 반영)**
- Day 1~6 결과를 제안서 요약 표로 추가
- “실행 준비 완료” → “Phase 0 proof-of-life 완료, Phase 1 착수 직전”

### B. 1주 우선순위 재정렬
기존 Phase 1 순서보다 아래 순서가 맞다.
1. stable relay (named tunnel or cloud deploy)
2. execution receipts + observability-lite
3. auth preflight + memory/embeddings reality check
4. Windows smoke-test
5. Cheap Team Mini / Handoff Mini
6. KR skill pack 확장

### C. 신규 레이어 명시
- Upgrade Guard / Known Good Versions
- Capability Audit (before/after config diff)
- Timeout Guard / Soft-cancel
- Browser Cleanup Guard

### D. 카카오 표현 수정
- “카카오 연동 완성” → “카카오 round-trip은 증명됨. 이제 과제는 relay 안정화와 복구력”
